Helix › Introduction
Helix
A configurable PID controller and chassis library for VEX V5 robots, built on the PROS framework.
VEX V5 PROS 3+ C++17 Open Source
🎛️
PID Control
Configurable kP, kI, kD with anti-windup, output clamping, and settle detection built in.
📐
Odometry
Track your robot's X, Y, and heading on the field using drive encoders, tracking wheels, and IMU.
📈
Motion Profiling
S-curve profiles limit jerk and acceleration for smooth, accurate movements without wheel slip.
📡
Telemetry Logger
Record PID states to SD card in real-time as CSV. Falls back to terminal if no SD card is present.
⚡
Async Movements
Non-blocking driveAsync() and turnAsync() let you do other work while driving.
🔩
PIDMotor
Wrap any motor or motor group with PID control and optional external encoder feedback.

Installation
PROS CLISetup

Add Helix to your PROS project by dropping the source files into the right directories. No package manager required.

Step 1 — Copy Headers

Copy the entire Helix/ folder from include/ into your project's include/ directory.

Bash
cp -r Helix/include/Helix your_project/include/Helix

Step 2 — Copy Source Files

Copy all .cpp files into src/Helix/ in your project.

Bash
cp -r Helix/src/Helix your_project/src/Helix

Step 3 — Include in main.cpp

C++
#include "Helix/Helix.hpp"

Step 4 — Build & Upload

Bash
pros build         # Compile
pros upload        # Upload to V5 Brain
pros terminal      # View serial output
📋 Requirements PROS 3.x or later, C++17 compiler support. The library targets VEX V5 hardware only.
Quick Start
Tutorial5 min

Get a working chassis up and running in under 5 minutes with these copy-paste examples.

Minimal Setup

This is the smallest possible Helix configuration — two motor groups, a chassis, and some movements.

C++ — main.cpp
#include "main.h"
#include "Helix/Helix.hpp"

using namespace Helix;

// --- Motors ---
pros::Motor leftFront (1, pros::E_MOTOR_GEAR_BLUE, true);
pros::Motor leftBack  (2, pros::E_MOTOR_GEAR_BLUE, true);
pros::Motor rightFront(3, pros::E_MOTOR_GEAR_BLUE, false);
pros::Motor rightBack (4, pros::E_MOTOR_GEAR_BLUE, false);

pros::Motor_Group leftSide ({leftFront, leftBack});
pros::Motor_Group rightSide({rightFront, rightBack});
pros::IMU imu(10);

// --- Chassis ---
Chassis::Config cfg;
cfg.drivetrain  = Drivetrain(&leftSide, &rightSide, 600, 3.25);
cfg.lateralPID  = PIDController(0.8, 0.001, 0.1);
cfg.turnPID     = PIDController(1.5, 0.002, 0.15);
cfg.imu         = &imu;
cfg.maxLateralSpeed = 100;
cfg.maxTurnSpeed    = 80;

Chassis chassis(cfg);

void initialize() {
    imu.reset();
    while (imu.is_calibrating()) pros::delay(10);
}

void autonomous() {
    chassis.drive(24);     // 24 inches forward
    chassis.turn(90);      // Turn 90° right
    chassis.drive(-12);   // 12 inches backward
    chassis.turnTo(0);     // Face heading 0
}

void opcontrol() {
    pros::Controller ctrl(pros::E_CONTROLLER_MASTER);
    while (true) {
        chassis.arcade(ctrl.get_analog(ANALOG_LEFT_Y),
                       ctrl.get_analog(ANALOG_RIGHT_X));
        pros::delay(20);
    }
}
✅ That's it! This is a complete, working Helix setup. Head to the Configuration pages to add odometry, motion profiling, and telemetry logging.
Motors & Drivetrain
Config Step 1Hardware

Declare your motors, group them, and build a Drivetrain struct that tells Helix about your robot's physical dimensions.

Step 1 Declare Motors

Each motor needs a port number, gear cartridge, and whether it's reversed. Find ports by going to Devices on the V5 Brain and unplugging each motor one at a time.

C++
// Left side — reversed so positive = forward
pros::Motor leftFront (11, pros::E_MOTOR_GEAR_BLUE, true);
pros::Motor leftMiddle( 3, pros::E_MOTOR_GEAR_BLUE, true);
pros::Motor leftBack  ( 1, pros::E_MOTOR_GEAR_BLUE, true);

// Right side — not reversed
pros::Motor rightFront(17, pros::E_MOTOR_GEAR_BLUE, false);
pros::Motor rightMiddle(10, pros::E_MOTOR_GEAR_BLUE, false);
pros::Motor rightBack  ( 8, pros::E_MOTOR_GEAR_BLUE, false);

Cartridge Reference

ConstantColorRPMUse when
E_MOTOR_GEAR_BLUEBlue600Speed-focused drivetrain (most common)
E_MOTOR_GEAR_GREENGreen200Torque-focused or heavy robot
E_MOTOR_GEAR_REDRed100Very heavy or highly geared mechanisms

Step 2 Group Motors

C++
pros::Motor_Group leftSide ({leftFront, leftMiddle, leftBack});
pros::Motor_Group rightSide({rightFront, rightMiddle, rightBack});

Step 3 Create Drivetrain

C++
Helix::Drivetrain drivetrain(
    &leftSide,    // left motor group
    &rightSide,   // right motor group
    600,          // RPM — must match your cartridge
    3.25          // wheel diameter in inches
);
📏 Measuring Wheel Diameter Common VEX wheel sizes: 4" omni = 4.0, 3.25" omni = 3.25, 2.75" omni = 2.75. Measure your actual wheel if it's worn — even 0.1" error compounds over distance.

Motion Profiling (Optional)

Enable S-curve profiling for smooth acceleration that reduces wheel slip.

C++
cfg.useMotionProfile = true;
cfg.maxJerk          = 40.0;   // Higher = smoother transitions
cfg.maxAcceleration  = 16.0;   // How fast you speed up
cfg.maxVelocity      = 60.0;   // Top speed

Feedforward (Optional)

Feedforward adds voltage based on the desired velocity and acceleration from the motion profile, reducing the load on the PID controller.

C++
// output = kV*vel + kA*accel + kS*sign(vel)
cfg.lateralkV = 0.12;   // Velocity feedforward
cfg.lateralkA = 0.002;  // Acceleration feedforward
cfg.lateralkS = 3.0;    // Static friction offset
cfg.turnkV    = 0.08;
cfg.turnkA    = 0.001;
cfg.turnkS    = 2.0;
⚠️ Start with feedforward at 0 Tune your PID first with all feedforward values at 0. Only add feedforward once your basic PID is working.
Odometry
Config Step 3Position Tracking

Track your robot's (X, Y, θ) position on the field using encoders and an IMU. Required for driveToPoint() and turnToPoint().

Basic Setup (Motor Encoders + IMU)

C++
Helix::Odometry::Config oCfg;
oCfg.leftMotors   = &leftSide;
oCfg.rightMotors  = &rightSide;
oCfg.imu          = &imu;
oCfg.wheelDiameter = 3.25f;
oCfg.trackWidth    = 12.0f;   // Distance between wheels (inches)

Helix::Odometry* odom = new Helix::Odometry(oCfg);
chassis.setOdometry(odom);

With External Tracking Wheels

External (passive) tracking wheels are more accurate than motor encoders because they don't slip under load.

C++
pros::ADIEncoder leftEnc ('A', 'B', false);
pros::ADIEncoder rightEnc('C', 'D', false);
pros::ADIEncoder horizEnc('E', 'F', false);

Helix::Odometry::Config oCfg;
oCfg.leftMotors        = &leftSide;
oCfg.rightMotors       = &rightSide;
oCfg.leftEncoder       = &leftEnc;     // Overrides motor encoders
oCfg.rightEncoder      = &rightEnc;
oCfg.horizontalEncoder = &horizEnc;   // Side-to-side tracking
oCfg.horizontalOffset  = -5.75f;    // Inches from center
oCfg.imu               = &imu;
oCfg.wheelDiameter     = 3.25f;

Using Odometry

C++
// Reset pose at start of autonomous
odom->setPose(Helix::Pose(0, 0, 90));   // x, y, heading

// Call update() every loop tick
void opcontrol() {
    while (true) {
        odom->update();
        Helix::Pose pose = odom->getPose();
        pros::lcd::print(0, "X:%.1f Y:%.1f H:%.1f",
                        pose.x, pose.y, pose.theta);
        pros::delay(20);
    }
}

Logger & Telemetry
Config Step 4SD Card

Record PID states, motor outputs, and odometry data to CSV for post-match analysis. Automatically writes to SD card when available, falls back to serial.

Basic Usage

C++
Helix::Logger logger;

void initialize() {
    logger.setLevel(Helix::LogLevel::INFO);
    logger.initSDCard();           // Check for SD card
    logger.info("System ready");
}

void autonomous() {
    logger.startRecording();       // Begin CSV capture
    // ... run movements ...
    logger.stopRecording();        // Flush & close file
}

Log Levels

LevelConstantUse for
DEBUGLogLevel::DEBUGVerbose tuning data
INFOLogLevel::INFOGeneral status messages
WARNLogLevel::WARNNon-fatal issues
ERRORLogLevel::ERRORFailures that affect operation

Recorded CSV Columns

CSV Header
timestamp, target, current, error, output, velocity,
leftMotor, rightMotor, x, y, theta

SD Card Behavior

Helix writes to /usd/log_XXXXX.csv on the SD card. If no SD card is inserted, data is buffered in memory (up to 1000 samples) and can be dumped to serial.

C++
// Dump buffered data to serial (if no SD card)
logger.dumpMotionData();

// Save buffered data to a specific file
logger.saveToSDCard("my_run.csv");

// Check how many samples were recorded
size_t count = logger.getMotionSampleCount();
💡 SD Card Tip Files are named by timestamp: log_00042.csv. Use a spreadsheet to plot the target vs current columns to visualize your PID response.

PIDController API
Reference

Complete reference for Helix::PIDController. Can be used standalone or embedded inside Chassis and PIDMotor.

Constructor

C++
PIDController(
    double kP,        // Proportional gain
    double kI,        // Integral gain (start at 0)
    double kD,        // Derivative gain
    double dt = 0.02  // Timestep in seconds (default 20ms)
);

Methods

MethodReturnsDescription
compute(setpoint, measurement)doubleCalculate PID output for this timestep. Call every 20ms.
reset()voidClear integral and previous error. Call before new movement.
setOutputLimits(min, max)voidClamp output to [-127, 127] range by default.
setIntegralLimit(limit)voidAnti-windup: cap integral accumulation.
setTolerance(tol, samples)voidSet settle detection threshold and sample count.
isSettled()boolReturns true if within tolerance for required consecutive samples.
getError()doubleCurrent error (setpoint - measurement).
getIntegral()doubleAccumulated integral term (useful for debugging windup).
getDerivative()doubleCurrent derivative (computed on measurement to avoid kick).

Preset Configs

C++
// Precise — slow but accurate
PIDController pid = Helix::PIDConfigs::PRECISE;    // 0.5, 0.0005, 0.05

// Balanced — good starting point
PIDController pid = Helix::PIDConfigs::BALANCED;   // 0.8, 0.001, 0.1

// Fast — aggressive, may overshoot
PIDController pid = Helix::PIDConfigs::FAST;       // 1.2, 0.001, 0.15
📝 Derivative on Measurement Helix computes the derivative on measurement (not error) to avoid "derivative kick" when the setpoint changes suddenly. This means kD always dampens, never amplifies.

Example — Arm Control

C++
pros::Motor armMotor(5, pros::E_MOTOR_GEAR_GREEN, false);
Helix::PIDMotor arm(&armMotor, Helix::PIDController(0.5, 0.001, 0.1));

void autonomous() {
    arm.setPID(90);                  // Move to 90°
    arm.waitUntilSettled();           // Block until done

    arm.setPIDRelative(45);          // Move +45° more
    if (arm.waitUntilSettled(2000)) {
        // Settled within 2 seconds
    }
}

MotionProfile API
Reference

Generate smooth S-curve trajectories that limit jerk, acceleration, and velocity to reduce wheel slip and overshoot.

The 7 Phases

An S-curve profile breaks motion into seven phases to transition through acceleration smoothly:

PhaseJerkDescription
1+JIncreasing acceleration
20Constant max acceleration
3−JDecreasing acceleration (approaching cruise)
40Cruise at max velocity
5−JBeginning deceleration
60Constant max deceleration
7+JEasing out of deceleration

Methods

MethodReturnsDescription
setConstraints(jerk, accel, vel)voidSet motion limits before generating.
generate(distance)boolCompute profile for given distance (can be negative).
calculate(time)TrajectoryPointGet position, velocity, accel at time seconds.
getDuration()doubleTotal profile duration in seconds.
isFinished(time)boolCheck if time >= getDuration().
reset()voidClear all profile state.

Built-in Presets

C++
// jerk=20, accel=8, vel=40 — precise/heavy robots
MotionProfile mp = Helix::MotionProfiles::CONSERVATIVE();

// jerk=40, accel=16, vel=60 — general use
MotionProfile mp = Helix::MotionProfiles::BALANCED();

// jerk=80, accel=32, vel=80 — light/fast robots
MotionProfile mp = Helix::MotionProfiles::AGGRESSIVE();
PID Tuning Guide
GuideEssential

Every robot is different. These steps will get you dialed in quickly using a systematic approach.

Tuning Order

Always tune in this order. Never add kI or feedforward until the basic PID is working.

#ParameterGoalSigns you're done
1kPGet close to targetRobot reaches target but oscillates
2kDStop oscillationRobot settles cleanly, no wobble
3kIFix steady-state errorRobot stops exactly on target (if needed)
4FeedforwardReduce PID workloadProfile tracks smoothly at all speeds

Step 1 — Tune kP

C++
// Start here. kI=0, kD=0
cfg.lateralPID = PIDController(0.5, 0, 0);

Run chassis.drive(24) and observe. Increase kP in steps of 0.1–0.2 until the robot oscillates around the target. Then back off by ~20%.

Step 2 — Tune kD

C++
// Add kD ≈ kP / 10 as starting point
cfg.lateralPID = PIDController(1.0, 0, 0.1);

Increase kD until oscillation stops. If robot moves sluggishly, kD is too high.

Step 3 — Tune kI (Only if needed)

C++
// Start very small — kI causes windup easily
cfg.lateralPID = PIDController(1.0, 0.001, 0.1);

// Set integral limit to prevent windup
chassis.getLateralPID().setIntegralLimit(500);
⚠️ kI Warning Most VEX robots don't need kI. Only add it if the robot consistently stops 1–2 inches short of the target even after tuning kP and kD.

Separate Lateral and Turn Tuning

Tune lateralPID and turnPID separately. Run drive() tests for lateral, turn() tests for angular. Typical starting ranges:

ControllerkP rangekD range
Lateral (drive)0.4 – 1.50.05 – 0.2
Turn1.0 – 3.00.1 – 0.4
💡 Use the Logger Start telemetry recording before each test run. Plot target vs current from the CSV to see exactly how the PID responds — overshoot, oscillation, and settle time are all immediately visible.
Autonomous Examples
GuideExamples

Copy-paste autonomous routines using Helix features.

Simple L-Shape Route

C++
void autonomous() {
    odom->setPose(Pose(0, 0, 90));

    chassis.drive(24);       // Forward 24 inches
    pros::delay(150);
    chassis.turn(-90);       // Turn left 90°
    pros::delay(150);
    chassis.drive(24);       // Forward 24 more
}

Async + Parallel Actions

C++
void autonomous() {
    // Start driving, don't block
    chassis.driveAsync(36, 80, 4000);

    // Do something else while driving
    arm.setPID(90);             // Raise arm simultaneously
    pros::delay(800);

    // Wait for both to finish
    chassis.waitUntilSettled();
    arm.waitUntilSettled();
}

Point-to-Point Navigation

C++
void autonomous() {
    odom->setPose(Pose(0, 0, 90));

    chassis.driveToPoint(24, 24);    // Go to (24", 24")
    chassis.turnToPoint(48, 0);      // Face (48", 0")
    chassis.driveToPoint(48, 0);     // Drive there
    chassis.driveToPoint(0, 0, 60);  // Return at 60% speed
}

Full Auton with Logging

C++
void autonomous() {
    odom->setPose(Pose(0, 0, 0));
    logger.startRecording();
    logger.info("Auton start");

    chassis.drive(24);
    chassis.turn(90);
    chassis.drive(24);
    chassis.turnTo(0);
    chassis.drive(-24);

    logger.stopRecording();
    logger.info("Auton complete");
}
